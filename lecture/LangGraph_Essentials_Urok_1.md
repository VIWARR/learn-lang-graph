# LangGraph Essentials

## Урок 1: Узлы и Состояния (Nodes & State)

---

### 🧠 1. Концептуальная база & Архитектура

Начнём с того, почему вы вообще здесь. Вы умеете писать `RunnableSequence` на LangChain: `prompt | llm | parser`. Это работает, пока ваша логика — прямая линия. Но как только появляется требование «если модель вернула низкую уверенность — переспроси», «если API упал — попробуй ещё раз», «если пользователь передумал — откати последний шаг», линейная цепочка превращается в паутину вложенных `if`, `try/except` и ручного управления состоянием через замыкания. Это не архитектура, это долг, который вы берёте под будущие проценты.

**LangGraph решает это, вводя явную модель конечного автомата (Finite State Machine).** Вместо последовательности вызовов у вас есть:

- **State** — единый, строго типизированный объект, который является *единственным источником правды* (single source of truth) на протяжении всего выполнения графа.
- **Node** — чистая функция (в идеале), которая принимает срез состояния и возвращает **частичное обновление** (partial update), а не мутирует его напрямую.
- **Edge** — правило перехода между узлами.

Ключевая ментальная модель, которую нужно усвоить прямо сейчас: **узел никогда не работает с "живым" объектом состояния напрямую**. Он получает копию-срез (снимок состояния на момент вызова) и возвращает `dict` с ключами, которые хочет изменить. Всё остальное LangGraph берёт на себя.

#### Что происходит "под капотом" при вызове `.compile()`

Это не формальность и не синтаксический сахар. Когда вы вызываете:

```python
graph = builder.compile()
```

LangGraph выполняет три критичных шага:

1. **Валидация топологии.** Проверяются "осиротевшие" узлы (без входящих рёбер), недостижимый `END`, отсутствие точки входа (`START`). Если граф структурно некорректен — вы получите исключение ещё до первого вызова, а не runtime-сюрприз в проде.
2. **Построение каналов состояния (Channels).** Каждое поле вашего `State` превращается во внутренний **Channel** — примитив, который знает, как принимать обновления. По умолчанию это `LastValue` (последнее значение побеждает), но если поле аннотировано редуктором (об этом — в Уроке 2), канал будет знать, как *сливать* новое значение со старым, а не просто перезаписывать его.
3. **Инъекция рантайм-зависимостей.** Именно на этапе `compile()` подключаются `checkpointer` (Урок 4) и точки прерывания `interrupt_before` / `interrupt_after` (Урок 5). Это значит, что архитектура персистентности — не надстройка поверх графа, а часть его скомпилированного рантайма.

**Почему это важно понимать уже на Уроке 1:** если вы спроектируете `State` небрежно — без чёткого разделения на поля "только для чтения", "аккумулирующиеся" и "перезаписываемые" — вы не сможете исправить это без **инвалидации всех сохранённых чекпоинтов**. Схема состояния — это фактически схема базы данных вашего агента. Меняется она так же дорого.

#### TypedDict vs Pydantic: когда что использовать

У вас есть два легитимных инструмента для описания схемы `State`:

| Критерий | `TypedDict` | Pydantic `BaseModel` |
|---|---|---|
| Скорость выполнения | Выше (нет валидации на каждый апдейт) | Ниже (валидация каждого присвоения) |
| Рантайм-валидация | Отсутствует — это просто подсказка типов | Полная, с кастомными валидаторами |
| Значения по умолчанию | Не поддерживаются нативно | Поддерживаются |
| Рекомендация | Внутренние графы, где данные уже проверены на входе | Графы, принимающие внешний непроверенный ввод (API, форма, LLM-вывод) |

**Инженерное правило:** если состояние графа — это первая линия обороны от "грязных" данных пользователя, берите Pydantic и валидируйте на входе в `State`. Если состояние — это внутренний рабочий буфер между уже провалидированными этапами, `TypedDict` дешевле и достаточно.

### 💻 2. Разбор кода (Production-ready пример)

Разберём реальный кейс: **узел приёма и первичной валидации заявки на кредитный скоринг**. Задача — принять сырые данные заявителя, строго провалидировать их и подготовить нормализованный объект для следующего узла (обращение к скоринговому API). Это первый узел будущего графа, который мы будем расширять на протяжении всего курса.

```python
"""
lending_intake.py

Первый узел графа кредитного скоринга: приём и валидация заявки.
Демонстрирует проектирование State через Pydantic, иммутабельное
обновление состояния и базовую сборку StateGraph.
"""

from __future__ import annotations

import asyncio
import logging
from datetime import date
from typing import Literal, Optional, TypedDict

from pydantic import BaseModel, Field, ValidationError, field_validator

from langgraph.graph import StateGraph, START, END

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s | %(levelname)s | %(name)s | %(message)s",
)
logger = logging.getLogger("lending_graph")


# --------------------------------------------------------------------------
# 1. Доменные модели: используем Pydantic, т.к. данные приходят "снаружи"
#    и должны быть провалидированы на границе системы.
# --------------------------------------------------------------------------

class LoanApplicant(BaseModel):
    """Сырые данные заявителя, полученные из внешней формы/API."""

    full_name: str = Field(min_length=2, max_length=120)
    birth_date: date
    requested_amount: float = Field(gt=0, le=5_000_000)
    monthly_income: float = Field(ge=0)
    national_id: str = Field(min_length=6, max_length=32)

    @field_validator("birth_date")
    @classmethod
    def applicant_must_be_adult(cls, value: date) -> date:
        age_years = (date.today() - value).days // 365
        if age_years < 18:
            raise ValueError("Заявитель должен быть совершеннолетним")
        return value


class ValidationVerdict(BaseModel):
    """Нормализованный результат первичной проверки."""

    is_valid: bool
    debt_to_income_hint: Optional[float] = None
    rejection_reason: Optional[str] = None


# --------------------------------------------------------------------------
# 2. Схема состояния графа.
#    Обратите внимание: State — это НЕ LoanApplicant. Это отдельная,
#    более широкая структура, которая будет расти по мере добавления узлов
#    (скоринг, решение андеррайтера и т.д. появятся в следующих уроках).
# --------------------------------------------------------------------------

class UnderwritingState(TypedDict):
    """Состояние графа андеррайтинга кредитной заявки."""

    raw_payload: dict                       # сырые данные с фронтенда, как пришли
    applicant: Optional[LoanApplicant]       # заполняется узлом intake_node
    verdict: Optional[ValidationVerdict]     # заполняется узлом intake_node
    status: Literal["pending", "validated", "rejected"]


# --------------------------------------------------------------------------
# 3. Узел графа.
#    Контракт узла: принял срез State -> вернул ЧАСТИЧНЫЙ dict с апдейтом.
#    Узел НИКОГДА не пишет в state["..."] напрямую.
# --------------------------------------------------------------------------

async def intake_validation_node(state: UnderwritingState) -> dict:
    """
    Валидирует сырые данные заявителя и готовит нормализованный объект
    для узла скоринга. Полностью изолирован: не имеет побочных эффектов
    на state, кроме возвращаемого dict.
    """
    logger.info("intake_validation_node: старт валидации заявки")

    try:
        applicant = LoanApplicant.model_validate(state["raw_payload"])
    except ValidationError as exc:
        logger.warning("Заявка отклонена на этапе валидации: %s", exc)
        return {
            "verdict": ValidationVerdict(
                is_valid=False,
                rejection_reason=f"Некорректные входные данные: {exc.error_count()} ошибок",
            ),
            "status": "rejected",
        }

    # Бизнес-правило: грубая прикидка DTI (debt-to-income) до похода во внешний скоринг.
    # Если заявитель просит сумму, кратно превышающую его годовой доход — не тратим
    # деньги на вызов платного скорингового API.
    annual_income = applicant.monthly_income * 12
    if annual_income <= 0:
        dti_hint = None
    else:
        dti_hint = round(applicant.requested_amount / annual_income, 2)

    if dti_hint is not None and dti_hint > 10:
        logger.info("Заявка отклонена по предварительному DTI-фильтру: %.2f", dti_hint)
        return {
            "applicant": applicant,
            "verdict": ValidationVerdict(
                is_valid=False,
                debt_to_income_hint=dti_hint,
                rejection_reason="Запрошенная сумма несоразмерна доходу заявителя",
            ),
            "status": "rejected",
        }

    logger.info("Заявка %s прошла первичную валидацию", applicant.national_id)
    return {
        "applicant": applicant,
        "verdict": ValidationVerdict(is_valid=True, debt_to_income_hint=dti_hint),
        "status": "validated",
    }


# --------------------------------------------------------------------------
# 4. Сборка графа.
# --------------------------------------------------------------------------

def build_graph():
    builder = StateGraph(UnderwritingState)
    builder.add_node("intake_validation", intake_validation_node)
    builder.add_edge(START, "intake_validation")
    builder.add_edge("intake_validation", END)
    return builder.compile()


async def main() -> None:
    graph = build_graph()

    initial_state: UnderwritingState = {
        "raw_payload": {
            "full_name": "Иван Петров",
            "birth_date": "1990-04-12",
            "requested_amount": 300_000,
            "monthly_income": 150_000,
            "national_id": "AB1234567",
        },
        "applicant": None,
        "verdict": None,
        "status": "pending",
    }

    try:
        result = await graph.ainvoke(initial_state)
    except Exception:
        logger.exception("Непредвиденный сбой при выполнении графа")
        raise

    logger.info("Итоговый статус заявки: %s", result["status"])
    logger.info("Вердикт: %s", result["verdict"])


if __name__ == "__main__":
    asyncio.run(main())
```

**На что обратить внимание в этом коде:**

- `intake_validation_node` **не бросает исключение** при бизнес-отказе (невалидные данные, плохой DTI) — это ожидаемая ветка бизнес-логики, а не сбой системы. `try/except` здесь ловит именно `ValidationError` от Pydantic, то есть техническую границу "мусор на входе". Настоящие системные сбои (недоступность БД, таймауты) в следующих уроках будут обрабатываться отдельно, через retry-политики и условную маршрутизацию на узел эскалации.
- Обратите внимание, что `applicant` и `verdict` объявлены как `Optional` — потому что при первом снимке состояния (`status: "pending"`) их ещё не существует. LangGraph не имеет проблем с постепенным "дозаполнением" состояния — это нормальный паттерн.

### 📊 3. Трассировка Состояния (State Lifecycle)

Проследим, как выглядит объект `UnderwritingState` на каждом шаге для нашего примера.

**Шаг 0 — состояние на входе в `graph.ainvoke()`:**

| Ключ | Значение |
|---|---|
| `raw_payload` | `{full_name: "Иван Петров", birth_date: "1990-04-12", ...}` |
| `applicant` | `None` |
| `verdict` | `None` |
| `status` | `"pending"` |

**Шаг 1 — вход в узел `intake_validation`:**
Узел получает **этот же снимок целиком** (не глубокую копию каждого объекта, а ссылку на текущее согласованное состояние на момент старта узла). Внутри узла выполняется `LoanApplicant.model_validate(...)` — данные проходят валидацию, DTI-фильтр пропускает заявку.

**Что возвращает узел (частичный update, а не весь State!):**

```python
{
    "applicant": LoanApplicant(full_name="Иван Петров", ...),
    "verdict": ValidationVerdict(is_valid=True, debt_to_income_hint=0.17),
    "status": "validated",
}
```

**Шаг 2 — слияние (merge) обновления с состоянием.**
Так как ни одно из полей `UnderwritingState` не аннотировано редуктором (это тема Урока 2), LangGraph применяет **дефолтный канал `LastValue`**: новое значение просто замещает старое для каждого ключа, присутствующего в returned dict. Ключ `raw_payload` в update отсутствует — значит, канал `raw_payload` **не трогается** и сохраняет исходное значение.

**Шаг 3 — состояние на выходе из узла (и на входе в `END`):**

| Ключ | Было | Стало | Механизм обновления |
|---|---|---|---|
| `raw_payload` | `{...}` | `{...}` (без изменений) | не возвращён в update → LastValue сохраняет старое |
| `applicant` | `None` | `LoanApplicant(...)` | LastValue → замена |
| `verdict` | `None` | `ValidationVerdict(is_valid=True, ...)` | LastValue → замена |
| `status` | `"pending"` | `"validated"` | LastValue → замена |

Это и есть весь "жизненный цикл" для линейного однодоузлового графа. Важно, что **на каждом супершаге (superstep) LangGraph пересобирает состояние целиком из каналов**, а не хранит "дельты". Именно поэтому чекпоинт (Урок 4) — это всегда консистентный полный снимок, а не патч поверх патча.

### ⚠️ 4. Ошибка новичка (Bad Practice vs Good Practice)

Самая частая архитектурная ошибка на старте — **прямая мутация объекта `state` внутри узла**, потому что "это же просто словарь, почему бы не поправить его на месте".

**❌ Bad Practice:**

```python
async def intake_validation_node(state: UnderwritingState) -> dict:
    applicant = LoanApplicant.model_validate(state["raw_payload"])

    # АНТИПАТТЕРН: мутируем входящий state напрямую
    state["applicant"] = applicant
    state["status"] = "validated"

    return state  # возвращаем весь мутированный объект
```

**Почему это ломается в проде, а не сразу на локальной машине:**

1. **Гонка данных при параллельном выполнении.** Как только у вас появится ветвление (`add_edge(START, "node_a")` и `add_edge(START, "node_b")` — параллельный супершаг, Урок 2), оба узла получат ссылку на один и тот же объект `state`, если он передаётся по ссылке без изоляции. Мутация "на месте" в одном узле может быть перезаписана мутацией в другом — классический race condition, который невозможно надёжно воспроизвести локально при последовательном тестировании.
2. **Разрушение механизма чекпоинтинга.** LangGraph ожидает **явный контракт**: узел возвращает *дельту*, которую канал применяет через свою логику слияния. Когда вы возвращаете весь мутированный `state`, вы фактически обходите систему каналов — LangGraph либо попытается слить весь объект как один update (что для `TypedDict` без редьюсера сработает "случайно правильно" только в однопоточном линейном графе), либо, при использовании кастомных редьюсеров на других полях, вы получите двойное применение логики слияния и трудноуловимые баги.
3. **Невозможность отладки через time-travel.** LangGraph позволяет "перемотать" граф к любому чекпоинту и посмотреть, что вернул конкретный узел. Если узел мутирует общий объект вместо возврата чистой дельты, история изменений теряет смысл — вы не сможете сказать, какой именно узел что изменил.

**✅ Good Practice:**

```python
async def intake_validation_node(state: UnderwritingState) -> dict:
    applicant = LoanApplicant.model_validate(state["raw_payload"])

    # Возвращаем ТОЛЬКО изменившиеся ключи. state остаётся нетронутым.
    return {
        "applicant": applicant,
        "status": "validated",
    }
```

**Правило, которое стоит вытатуировать себе на предплечье:** узел читает `state`, как `Mapping` (только для чтения), и общается с графом исключительно через `return`-значение. Если вам физически хочется написать `state["x"] = ...` — это сигнал, что вы либо не понимаете модель данных LangGraph, либо пытаетесь обойти архитектуру, а не работать с ней.

### 🛠️ 5. Лабораторная мини-работа

**Задание:**

Возьмите граф `lending_intake.py` из раздела 2 и перепроектируйте схему состояния, добавив поддержку **повторной подачи заявки после отказа** (re-submission flow). Требования:

1. Добавьте в `UnderwritingState` поле `attempt_number: int`, которое должно корректно инициализироваться при первом запуске и **вручную** (без редьюсеров — они впереди, в Уроке 2) увеличиваться на 1 при каждом повторном вызове `intake_validation_node`, если предыдущий `status` был `"rejected"`.
2. Добавьте новую Pydantic-модель `RejectionHistoryEntry` (причина отказа + номер попытки) и поле `rejection_history: list[RejectionHistoryEntry]` в состояние. При каждом отказе новая запись должна **добавляться** к истории, а не затирать предыдущую.
3. Ограничьте число попыток тремя: на 4-й неудачной попытке узел должен вернуть `status: "rejected"` с флагом `rejection_reason="Превышено число попыток подачи заявки"`, независимо от результата валидации данных.

**Вектор решения (Подсказка):**

Обратите пристальное внимание на пункт 2. Если вы напишете что-то вроде:

```python
return {"rejection_history": [new_entry]}
```

— вы, скорее всего, получите список из **одного** элемента при каждом вызове, потому что без редьюсера сработает `LastValue` и полностью **перезапишет** старый список, а не добавит к нему запись. Это ровно та ловушка, которую мы разбирали в разделе 4 — только на этот раз не через прямую мутацию, а через незнание, что *по умолчанию списки в LangGraph не аккумулируются сами по себе*. Чтобы честно решить задачу средствами Урока 1, вам придётся внутри самого узла **вручную прочитать** `state["rejection_history"]`, сконструировать **новый** список (`старый_список + [новая_запись]`) и вернуть его целиком как обновлённое значение поля. Это неэлегантно и многословно — и именно эту боль в следующем уроке снимут функции-редукторы (`operator.add`, кастомные merge-функции). Прочувствуйте проблему руками, прежде чем получить инструмент, который её решает.
